# Fountainhead — Tom Geldschläger Website

Premium website for Fountainhead (Tom Geldschläger), a Berlin-based guitarist, producer, and mixing engineer.

## Stack

- **Framework**: Next.js 15 (App Router) + React 19
- **Language**: TypeScript 5.9 (strict)
- **Styling**: Tailwind CSS v4 + Radix UI + shadcn/ui (new-york), Framer Motion
- **Auth**: NextAuth (Auth.js) v5 — JWT sessions, admin + client-portal
- **Database**: PostgreSQL + Drizzle ORM (`drizzle-orm/node-postgres`)
- **Data fetching**: Server Components query Drizzle directly; client interactivity uses TanStack Query hooks generated by Orval (`@workspace/api-client-react`)
- **State**: Zustand (cart, persisted to localStorage)
- **Forms**: React Hook Form + Zod
- **Package manager**: pnpm workspaces (Node 24)

### Server vs Client Components

- Pages that only display data → **Server Components** fetching via Drizzle directly.
- Pages with cart/forms/auth/hooks → **Split**: `page.tsx` (Server, fetches via Drizzle, passes props) + `*-client.tsx` (Client, owns interactivity).
- `"use client"` is added only where a component uses hooks, events, browser APIs, or client-only libraries.

### Auth (NextAuth v5)

- **Admin**: Credentials (`admin-credentials`) — `admin_users` + bcrypt. Signs in at `/tomfountainhead-admin/login`.
- **Portal**: Google (primary — auto-creates a `portal_users` row on granted access) + Credentials (`portal-credentials`) for seeded accounts. Both gated on `portal_access.status === "granted"`.
- `middleware.ts` protects `/tomfountainhead-admin/*`, `/api/admin/*`, `/portal`, and `/api/portal/{lessons,resources}` by `token.role` (admin vs portal, never interchangeable). `/tomfountainhead-admin/login`, `/api/auth/*`, and `/api/portal/access` stay public.

## Running locally

```bash
pnpm install
pnpm dev          # next dev
pnpm build        # typecheck + next build
pnpm typecheck    # tsc (libs + app + scripts)
pnpm push         # drizzle-kit push (apply schema, incl. portal_users)
pnpm seed         # seed content
```

Create a `.env` at the repo root (see `.env.example`):

```
DATABASE_URL=postgresql://...
AUTH_SECRET=<openssl rand -base64 32>
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
```

> Local builds on **Windows** are limited: `pnpm-workspace.yaml` intentionally excludes non-Linux native binaries (lightningcss/oxide/esbuild) to keep installs small for Vercel. `pnpm dev`, `pnpm typecheck`, and the Vercel build work; a full local `next build` on Windows needs those win32 binaries un-excluded.

## Deploying to Vercel

1. **Import the repo** into Vercel. It auto-detects Next.js at the root (no `vercel.json` needed). Install command: `pnpm install`; build command: `pnpm build` (or `next build`).
2. **Environment variables** (Project → Settings → Environment Variables — set for Production/Preview, available at build time too):
   - `DATABASE_URL` — Postgres connection string (e.g. Supabase). **Required at build time** (Server Components import the DB module).
   - `AUTH_SECRET` — `openssl rand -base64 32`.
   - `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` — from Google Cloud Console.
3. **Apply the schema** once the DB is reachable: `pnpm push` (creates `portal_users` and syncs the rest), then `pnpm seed` if seeding content.
4. **Google OAuth**: in Google Cloud Console add the authorized redirect URI
   `https://<your-domain>/api/auth/callback/google` (and the Vercel preview URL if needed).
5. NextAuth auto-detects the deployment URL (`trustHost: true`); no `AUTH_URL` needed on Vercel.
